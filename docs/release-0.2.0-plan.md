# 0.2.0 需求清单

这份文档的目标不是继续发散想法，而是把 `Package Vision 0.2.0` 收敛成一份真正可执行的版本清单。

> 截至 `2026-04-07`，这份清单里的 3 个 P0 已经在 `main` 分支实现完成，并已作为 `0.2.0` 正式发布到 Marketplace。

当前 `0.1.0` 已经完成了这些核心能力：

- 在 Activity Bar 提供独立入口
- 展示一个或多个 `package.json` 中的依赖
- 查询最新版本并显示状态
- 支持单个依赖升级
- 支持大版本升级策略
- 支持状态筛选
- 支持 `npm`、`pnpm`、`yarn`、`bun`

所以 `0.2.0` 不再追求“把基础能力补齐”，而是重点提升可用性。

## 1. 版本主题

`0.2.0` 的推荐主题是：

**从“单个依赖查看与升级”升级到“项目级依赖整理与批量处理”。**

换句话说，`0.1.0` 解决的是“我能不能在 VS Code 里处理一个包”；`0.2.0` 要解决的是“当一个项目里有很多依赖时，我能不能更快找到重点并批量处理”。

## 2. 版本目标

`0.2.0` 建议聚焦这 3 个结果：

1. 用户能更快定位需要关注的依赖
2. 用户能对一批依赖执行安全升级，而不是只能一个一个点
3. 在 monorepo 场景里，用户能看出同一个依赖是否存在版本分裂

## 3. 成功标准

如果 `0.2.0` 做完后满足下面几点，就说明这个版本方向是对的：

- 在依赖较多的项目里，用户能通过搜索和筛选快速缩小范围
- 用户能在不打开终端的情况下完成一次“批量保守升级”
- monorepo 用户能快速发现同一个依赖在多个子包中声明不一致
- 不破坏当前单包升级、筛选、版本范围设置和多包管理器支持

## 4. P0 功能范围

下面 3 项建议作为 `0.2.0` 的 P0，也就是“本版本真正要交付的东西”。

### 4.1 依赖搜索与组合筛选

当前状态：已实现，待发布

#### 为什么先做这个

当前已经支持按状态筛选，但还缺少“按包名快速定位”的能力。对于依赖很多的项目，搜索会比新增更多状态按钮更有用。

#### 用户故事

作为一个用户，我希望在 `Package Vision` 里快速搜索某个依赖名，并且和现有的状态筛选组合使用，这样我能更快定位目标包。

#### 建议范围

- 在视图标题栏新增 `Search Dependencies`
- 支持输入包名关键字进行模糊匹配
- 搜索和当前状态筛选可以同时生效
- 有搜索条件时，视图标题或描述中展示当前关键字
- 提供 `Clear Search` 入口

#### 验收标准

- 输入 `react` 时，只显示名称包含 `react` 的依赖
- 搜索结果能和 `Outdated`、`Lookup Failed` 等筛选一起使用
- 搜索无结果时，显示明确空状态
- 清空搜索后，列表恢复到原始筛选结果

#### 学习价值

- 学习如何在 `TreeDataProvider` 外维护额外的视图状态
- 学习 VS Code 命令、Quick Pick、视图标题按钮之间的配合

#### 预计会影响的文件

- `package.json`
- `src/extension.ts`
- `src/views/dependencyTreeProvider.ts`
- `src/views/dependencyFilterUtils.ts`
- `tests/`
- `src/test/suite/`

### 4.2 Monorepo 版本分裂检测

当前状态：已实现，待发布

#### 为什么值得做

当前已经能扫描多个 `package.json`，但还不能回答一个很现实的问题：

“同一个依赖在不同子包里是不是已经写成了不同版本？”

这类问题在 monorepo 里很常见，而且很适合做成 `Package Vision` 的差异化能力。

#### 用户故事

作为一个 monorepo 用户，我希望知道同一个依赖是否在多个 package 中声明了不同版本，这样我能尽早发现版本分裂。

#### 建议范围

- 聚合同名依赖在多个 package 中的声明版本
- 当同名依赖出现多个声明版本时，标记为“版本分裂”
- 在 tooltip 中展示冲突来源，例如哪些 package 正在使用哪些版本
- 新增一个 `Version Drift` 筛选

#### 验收标准

- 当 `react` 在不同 package 中使用不同声明版本时，相关条目都会显示分裂提示
- 单 package 项目不会错误触发分裂状态
- `Version Drift` 筛选只展示存在分裂的依赖
- tooltip 中能看出冲突来自哪些 package

#### 学习价值

- 学习如何在现有数据模型上做“跨 package 聚合分析”
- 学习如何在不破坏原有状态系统的前提下新增一个横切能力

#### 预计会影响的文件

- `src/models/dependency.ts`
- `src/services/packageJsonService.ts` 或新增聚合 util
- `src/views/dependencyTreeProvider.ts`
- `src/views/dependencyFilterUtils.ts`
- `tests/`

### 4.3 批量保守升级

当前状态：已实现，待发布

#### 为什么这是 `0.2.0` 的主功能

从用户价值看，这是最能明显提升可用性的能力。

当前插件已经能升级单个依赖，但很多真实场景并不是“只改一个包”，而是：

- 先筛出一批过时包
- 尽量在当前 major 内做安全升级
- 一次处理掉

#### 用户故事

作为一个用户，我希望对当前列表中可安全升级的依赖执行一次批量升级，这样我不用重复点很多次单包升级。

#### 建议范围

- 在视图标题栏新增 `Upgrade Safe Dependencies`
- 默认只处理当前可见列表中的依赖
- 只升级“当前 major 内存在更高安全版本”的依赖
- 如果某个依赖只有更高 major 没有安全目标，则跳过
- 升级前给出确认框，说明：
  - 将处理多少个依赖
  - 会影响哪些 package
  - 不会默认执行 major upgrade
- 升级过程中显示总进度
- 最终给出成功 / 失败 / 跳过数量汇总

#### 验收标准

- 在 `Outdated` 或其他筛选条件下，批量升级只作用于当前可见且可安全升级的依赖
- 批量升级不会默认触发 latest major
- 某个依赖升级失败时，其他依赖仍可继续执行
- 执行结束后自动刷新视图
- 输出面板能看到每个依赖的执行日志

#### 学习价值

- 学习批量命令编排
- 学习长任务的进度展示和结果汇总
- 学习如何在单包升级服务之上复用已有能力，而不是重写一套流程

#### 预计会影响的文件

- `package.json`
- `src/extension.ts`
- `src/services/packageManagerService.ts`
- `src/services/upgradeStrategyUtils.ts`
- `src/views/dependencyTreeProvider.ts`
- `tests/`
- `src/test/suite/`

## 5. P1 可选项

这些功能很有价值，但更适合放在 `0.2.x` 或 `0.3.0`，不要和上面的 3 个 P0 混在一起。

- 显示更接近 `npm outdated` 的 `current / wanted / latest`
- 忽略规则，例如忽略某个包或忽略 major upgrade
- `peerDependencies` 风险提示
- 依赖详情页或 changelog / release notes 入口
- `package.json` 内的 CodeLens

## 6. 当前明确不做

为了让 `0.2.0` 真正可落地，建议先不做这些：

- Webview 仪表盘
- 安全漏洞扫描整合
- 自动解析 breaking changes
- 自动抓取并聚合 changelog 内容
- 一键批量升级到最新 major

## 7. 推荐开发顺序

推荐按这个顺序推进：

1. 依赖搜索与组合筛选
2. Monorepo 版本分裂检测
3. 批量保守升级
4. 文档、截图、测试补齐

这样安排的原因是：

- 搜索功能最轻，适合先把额外视图状态模型建立起来
- 版本分裂检测会强化数据聚合能力
- 批量升级是最重的交互流程，放在后面更稳

## 8. 测试与验证建议

为了让 `0.2.0` 可控，建议每个 P0 都配上对应测试：

- 单元测试
  - 搜索匹配规则
  - 版本分裂判断逻辑
  - 批量升级候选集选择逻辑
- Extension Host 集成测试
  - 新命令是否注册成功
  - 搜索和筛选能否共同生效
  - 版本分裂在 fixture 工作区里能否正确出现
- 手工冒烟测试
  - 单 package 项目
  - monorepo 项目
  - `npm` / `pnpm` / `yarn` / `bun` 至少各覆盖一个基本场景

## 9. 完成定义

`0.2.0` 发布前，建议至少满足下面这些条件：

- 3 个 P0 功能全部可用
- `npm run check:full` 通过
- README 和截图更新到新能力
- CHANGELOG 更新到 `0.2.0`
- VSIX 本地打包和安装验证通过
- Marketplace 描述同步新卖点

## 10. 一句话版本定义

如果要把 `0.2.0` 用一句话说清楚，我建议是：

**让 Package Vision 从“查看和升级单个依赖”，进化成“在 VS Code 中快速整理一整个项目的依赖”。**
