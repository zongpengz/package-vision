# 项目复盘文档

这份复盘不是为了记录“我们做了多少功能”，而是为了把这次项目真正学到的东西沉淀下来。对新手来说，这类复盘很有价值，因为它能把“做完了一个插件”和“真正理解了插件开发”区分开。

## 1. 这次项目真正完成了什么

从结果看，`Package Vision` 已经不是一个练手级 Hello World，而是一个完整走过一轮产品闭环的 VS Code 扩展：

- 有明确的产品入口和用户场景
- 有 Activity Bar 入口和 Tree View UI
- 能读取真实工作区中的 `package.json`
- 能查询最新版本并展示状态
- 能升级单个依赖
- 支持 `npm`、`pnpm`、`yarn`、`bun`
- 支持 monorepo / 多个 `package.json`
- 有设置项、筛选、日志和状态反馈
- 有单元测试和 Extension Host 集成测试
- 能打包 VSIX
- 已经发布到 VS Code Marketplace

这件事的学习价值在于：你不是只学了某一个 API，而是完整经历了“需求 -> 设计 -> 开发 -> 调试 -> 测试 -> 打包 -> 发布”。

## 2. 你真正学到的第一件事：VS Code 插件不是“普通前端项目”

这是这次项目最重要的认识之一。

一开始很容易把 VS Code 插件理解成“做一个嵌在编辑器里的前端页面”，但这个项目会让你看到，实际并不是这样。一个典型的 VS Code 扩展至少有三层：

- 扩展清单：`package.json`
- 扩展宿主逻辑：`src/extension.ts` 和服务层
- 界面承载方式：Tree View、命令、菜单、状态栏，必要时才是 Webview

你这次做的主体其实是：

- Node Extension
- VS Code 原生视图能力
- 不是 React 单页应用

这会直接改变你对“插件开发”的理解：很多时候你不是在“画页面”，而是在“把能力挂到 VS Code 的扩展点上”。

## 3. 你学到的第二件事：先收敛问题，比一开始做大更重要

如果一开始就把目标写成：

- 支持所有包管理器
- 支持 monorepo
- 支持批量升级
- 支持各种复杂筛选
- 支持炫酷 UI

这个项目大概率会失控。

你这次真正做对的一步，是先把第一版收敛成：

- 左侧栏入口
- 展示依赖
- 查询最新版本
- 升级单个包

然后再一步一步扩展出：

- 更好的状态展示
- 更多包管理器
- monorepo
- 设置项
- 筛选
- 测试
- 发布

这背后的经验很重要：对于工程项目来说，真正推动你成长的不是“第一次就设计得很大”，而是“把一个可验证的最小链路跑通，然后持续增强”。

## 4. 你学到的第三件事：`package.json` 在扩展里是“系统入口”

普通 Node 项目里，`package.json` 更多像依赖清单和脚本入口。  
在 VS Code 扩展里，它还是：

- 激活条件声明
- 视图入口声明
- 命令声明
- 菜单声明
- 设置项声明
- Marketplace 元数据来源

也就是说，它既是技术文件，也是产品配置文件。

你通过这个项目会逐渐建立一个很关键的意识：

- 扩展能不能被正确激活
- 左侧入口会不会出现
- 右键菜单会不会出现
- 商店页面长什么样

这些都不是“运行时代码自己决定的”，而是大量由 `package.json` 决定的。

## 5. 你学到的第四件事：VS Code 原生 UI 很强，先不要急着上 Webview

这个项目现在主要用了：

- `Activity Bar`
- `View Container`
- `Tree View`
- `commands`
- `menus`
- `ThemeIcon`

这套原生能力已经足够承载一个很好用的依赖管理视图了。

这里最大的学习点是：  
不是每个插件都需要复杂前端。很多高质量扩展的第一版，都是先把原生能力吃透。

你这次也会感受到原生能力的几个好处：

- 和 VS Code 风格一致
- 交互成本低
- 不需要自己处理太多 UI 基础设施
- 调试和维护成本更低

这也是为什么这次没有走 Webview 路线，而是先用 Tree View 把核心价值做出来。

## 6. 你学到的第五件事：一个好扩展，代码结构必须先分层

这次代码并不是把所有逻辑都塞进 `extension.ts`，而是逐渐拆成了：

- 入口层：扩展激活、命令注册
- 视图层：Tree Provider、筛选状态
- 数据层：读取 `package.json`
- registry 层：查询最新版本
- 包管理器层：构造并执行升级命令
- 纯逻辑层：版本比较、命令构造、版本范围格式化

这种分层最直接的好处有两个：

- 功能更容易继续扩展
- 纯逻辑更容易测试

这次后面之所以能比较自然地补单元测试，就是因为很多逻辑已经被拆成了不依赖 VS Code 运行时的纯函数模块。

这会让你以后做项目时更容易判断：

- 哪些代码应该依赖框架
- 哪些代码应该尽量独立出来

## 7. 你学到的第六件事：调试不是附属动作，而是主流程的一部分

在这个项目里，调试不是“写完后顺手看一眼”，而是整个开发过程的主线之一。

你已经接触到了几个最关键的调试概念：

- 按 `F5` 启动 `Extension Development Host`
- 在源码窗口打断点
- 在扩展宿主里触发真实行为
- 借助 `Output` 面板看日志
- 遇到视图声明变更时执行 `Developer: Reload Window`

这和一般网页前端开发最大的区别之一是：

- 你不是在浏览器里刷新页面
- 你是在“另一个 VS Code 实例”里验证你的扩展

一旦你把这个调试心智建立起来，后面做命令、视图、保存事件监听、状态刷新时就会轻松很多。

## 8. 你学到的第七件事：用户体验优化，往往比功能本身更能拉开差距

这个项目有几个特别典型的体验增强点：

- 升级中的 loading 状态
- 彩色状态图标
- 更明确的成功和失败反馈
- 输出面板入口
- 快速筛选
- 设置项控制版本范围风格

这些东西从“功能定义”上看不是核心能力，但从真实使用体验上看，它们很关键。

这会让你学到一个成熟项目很常见的判断方式：

- 第一层是“能不能做”
- 第二层是“用起来清不清楚”

很多时候，一个插件是否显得专业，取决于第二层。

## 9. 你学到的第八件事：看起来简单的“升级依赖”，背后其实很复杂

一开始，“点击按钮升级依赖”听起来像一个简单动作。  
但这次实现之后你应该已经看到，真正复杂的是边界：

- `npm`、`pnpm`、`yarn`、`bun` 命令不一样
- monorepo 里命令执行目录不一样
- workspace 模式下参数不一样
- `dependencies` 和 `devDependencies` 参数不一样
- 升级后是否保留 `^`、`~`、精确版本也不一样

这个过程的学习价值很大，因为它会让你逐渐意识到：

- 真正的工程复杂度，很多时候来自边界情况
- 一个功能一旦走到“真实可用”，就一定会牵涉环境差异和状态差异

也正因为如此，这个项目已经不只是“会不会调用一个命令”，而是“会不会设计一层能消化差异的抽象”。

## 10. 你学到的第九件事：monorepo 支持不是“顺手加一下”

monorepo 这部分的学习价值非常高，因为它会逼着你面对“工作区”和“单项目”的区别。

当项目只有一个 `package.json` 时，很多逻辑都可以偷懒。  
但一旦支持多个 package，就必须重新思考：

- 怎么扫描 manifest
- 怎么表示 package 身份
- Tree View 怎么分组
- 升级动作到底作用于哪个 package
- 不同包管理器在 workspace 下怎么定位目标包

这部分能让你真正接触到“项目规模一上来，数据模型就必须升级”。

## 11. 你学到的第十件事：测试不是最后补作业，而是帮助你继续迭代

你这次已经接触到了两类测试：

- 单元测试
- Extension Host 集成测试

而且你应该已经感受到它们的分工：

- 单元测试负责纯逻辑
- 集成测试负责 VS Code 运行时和主链路

更重要的是，你还真实遇到了一个非常工程化的问题：

- 集成测试 runner 并不是永远稳定可用

这件事本身也很有学习价值，因为它会让你建立更现实的测试观：

- 测试是重要的
- 但测试基础设施本身也可能出问题
- 所以要区分“业务功能坏了”还是“测试框架链路坏了”

这比“单纯把测试都写绿”更接近真实工程环境。

## 12. 你学到的第十一件事：发布不是最后一条命令，而是一整套交付能力

从这次首次上架你已经接触到了完整的发布链路：

- `vsce package`
- VSIX 本地安装验证
- Marketplace metadata
- icon
- README
- screenshots
- changelog
- Azure DevOps PAT
- publisher
- `vsce publish`

很多初学者会把“发布”理解成最后执行一条命令。  
但这次项目应该已经让你看到：

- 发布页面是什么样
- 图标和截图是否清楚
- README 是否像产品介绍
- 用户能否理解这个插件值不值得装

这些都属于发布的一部分。

所以更准确地说，你这次学到的是：

- 不是“怎么上传一个扩展”
- 而是“怎么把一个扩展作为产品交付出去”

## 13. 你这次踩到的几个很值得记住的坑

这几个坑以后会反复帮到你：

### 13.1 不是每条官方链路都会在你的本机环境里完全顺滑

例如这次 Extension Host 测试 runner 的 CLI 启动问题，就说明：

- 文档是必要的
- 但本机环境差异仍然可能让链路出现意外

### 13.2 商店发布成功，不代表公开页面立刻完全稳定

你已经看到：

- `vsce publish` 成功
- `vsce show` 能查到扩展
- 但公开页面一开始可能还有传播延迟或缓存影响

这会提醒你以后发布后不要只看一个信号。

### 13.3 认证和发布平台的门槛，往往不在代码里

例如这次遇到的：

- Azure DevOps organization
- PAT
- publisher
- 组织上限报错

这些都不是编码问题，但它们会决定你能不能真正把产品发出去。

## 14. 如果把这次项目浓缩成一句方法论

可以浓缩成这句话：

先跑通最小闭环，再逐步增强；每增加一个真实能力，就顺手补文档、验证和发布意识。

这是一个非常值得你带到下一个项目里的工作方法。

## 15. 你下一阶段最值得继续学什么

如果你想继续用这个项目提高能力，我建议优先顺序是：

1. 解决 `check:full` 里的 Extension Host 集成测试问题  
   这会提升你对工具链和自动化测试的理解。

2. 做一次常规版本迭代发布  
   比如从 `0.0.1` 到 `0.0.2`，你会真正熟悉 changelog、版本号和发布节奏。

3. 继续优化 Marketplace 展示  
   例如更精炼的中英文介绍、截图顺序、下一版 release notes。

4. 选择一个“边界功能”继续扩展  
   比如批量升级、忽略包、更多筛选条件、私有 registry 处理。

## 16. 最后给你的结论

如果只用一句话评价这次项目：

你已经不是“体验过 VS Code 扩展开发”了，而是已经完整做过一次真实的 VS Code 扩展交付。

这两者差别很大。

前者只是知道扩展大概怎么写，后者意味着你已经接触过：

- 需求拆分
- API 选型
- 运行时调试
- 边界处理
- 测试分层
- 打包验证
- 商店发布

这就是这次项目最有学习价值的地方。
