# 产品需求文档

## 1. 项目背景

前端项目经常需要检查依赖包是否过时。常见做法是：

- 打开终端执行 `npm outdated`
- 结合 `package.json` 手工判断要不要升级
- 再执行安装命令完成更新

这个流程的问题是：

- 信息分散在终端和编辑器之间
- 对新手不够直观
- 升级动作和查看动作是割裂的
- 当项目依赖很多时，人工筛选成本较高

Package Vision 想解决的是“在 VS Code 里完成依赖版本巡检和局部升级”这件事。

## 2. 产品定位

Package Vision 是一个面向前端开发者的 VS Code 扩展。它提供一个位于 Activity Bar 的独立入口，在侧边栏中展示项目依赖的版本状态，并允许开发者对单个依赖执行升级操作。

它不是一个包管理器替代品，而是一个依赖可视化和升级入口增强工具。

## 3. 目标用户

- 维护 React、Vue、Vite、Next.js、Nuxt 等前端项目的开发者
- 想减少终端切换频率的开发者
- 希望通过真实项目学习 VS Code 扩展开发的新手

## 4. 核心问题

用户希望在不离开 VS Code 的情况下，快速知道：

- 当前项目有哪些依赖
- 每个依赖在 `package.json` 中声明的版本是什么
- npm registry 上的最新稳定版本是什么
- 哪些包值得升级
- 如何只升级一个特定包，而不是一次性升级全部

## 5. 目标与非目标

### 5.1 产品目标

- 在编辑器左侧提供清晰稳定的功能入口
- 让依赖版本状态一眼可见
- 让单个依赖升级动作可点击、可确认、可反馈
- 让实现过程适合新手逐步学习和扩展

### 5.2 MVP 非目标

第一版暂不追求以下能力：

- 自动分析 breaking changes
- 自动阅读 changelog 或 release notes
- 一次升级全部依赖
- 完整支持 monorepo 多包场景
- 完整支持所有包管理器的所有细节
- 对私有 registry、代理、认证场景做深度适配

## 6. 名词定义

- 声明版本：`package.json` 中写下的版本范围，例如 `^18.3.1`
- 最新版本：注册表中 `latest` dist-tag 指向的最新稳定版本
- 过时：最新版本不再被当前声明版本覆盖，或当前声明版本明显落后
- 升级：执行包管理器命令并更新 `package.json` 与锁文件

说明：为了降低第一版复杂度，本项目的 MVP 先关注“声明版本”和“最新版本”。如果以后要更接近 `npm outdated`，可以再增加“已安装版本”和“wanted 版本”。

## 7. MVP 功能范围

### 7.1 必做功能

- 在 Activity Bar 中增加 `Package Vision` 入口
- 识别当前工作区根目录下的 `package.json`
- 解析以下依赖字段：
  - `dependencies`
  - `devDependencies`
- 在侧边栏中按分组展示依赖列表
- 每个依赖展示：
  - 包名
  - 声明版本
  - 最新版本
  - 状态标识
- 支持手动刷新版本信息
- 支持升级单个依赖到最新版本
- 升级成功后自动刷新视图

### 7.2 推荐但可延后

- 支持 `peerDependencies`
- 支持 `optionalDependencies`
- 支持右键菜单升级
- 支持顶部工具栏刷新
- 支持空状态提示和引导文案
- 支持错误提示和日志查看

### 7.3 后续迭代

- 支持 `升级全部`
- 支持筛选“仅显示过时依赖”
- 支持显示 changelog 链接
- 支持 monorepo 多个 `package.json`
- 支持 pnpm、Yarn、bun 的深度适配
- 支持保存升级策略设置，例如 `^`、`~`、精确版本

## 8. 用户故事

### 8.1 查看依赖状态

作为一个前端开发者，我希望打开左侧 `Package Vision` 之后，就能看到当前项目依赖的声明版本和最新版本，这样我不用来回切终端。

### 8.2 刷新版本信息

作为一个用户，我希望点击刷新按钮后重新获取最新版本信息，这样我能确认列表不是旧数据。

### 8.3 升级单个依赖

作为一个用户，我希望对某个依赖直接点击升级，这样我能局部更新依赖，而不是一次性修改整个项目。

### 8.4 无项目时得到反馈

作为一个用户，如果当前工作区没有 `package.json`，我希望看到明确提示，而不是一个空白视图。

## 9. 交互要求

### 9.1 入口位置

- 入口必须出现在 Activity Bar
- 视觉上与 Explorer、Search、Extensions 同级
- 点击后在主侧边栏打开扩展视图

### 9.2 依赖列表

建议第一版采用 Tree View：

- 第一层：依赖分组
- 第二层：具体包项

每个包项建议展示：

- `label`：包名
- `description`：当前声明版本
- `tooltip`：当前版本、最新版本、状态、升级命令提示

### 9.3 升级动作

升级动作建议通过以下一种或多种方式暴露：

- 包项右键菜单中的“升级到最新版本”
- 包项点击后的命令
- 视图工具栏动作

升级前建议弹出确认框，至少说明：

- 包名
- 当前声明版本
- 即将升级到的目标版本
- 可能修改 `package.json` 和锁文件

## 10. 功能验收标准

### 10.1 入口与视图

- 安装扩展后，Activity Bar 中能看到扩展图标
- 点击后能打开 `Package Vision` 视图

### 10.2 数据展示

- 工作区存在 `package.json` 时能显示依赖
- 每个依赖能看到包名、声明版本、最新版本
- 过时依赖有清晰可辨识的状态

### 10.3 刷新能力

- 执行刷新命令后，界面能重新获取最新版本
- 失败时有错误提示，不会导致视图失效

### 10.4 升级能力

- 对单个依赖执行升级后，命令能被正确调用
- 升级成功后，`package.json` 或锁文件发生预期变化
- 视图自动刷新为最新状态

### 10.5 空状态与异常

- 没有工作区时显示提示
- 没有 `package.json` 时显示提示
- 网络失败时显示错误信息
- 包管理器不可用时显示明确说明

## 11. 约束与风险

### 11.1 版本语义风险

包管理器默认保存策略可能不是“精确版本”。例如 npm 默认常见行为是写入带 `^` 的范围，所以“升级到最新版本”在 `package.json` 中可能表现为 `^x.y.z`，而不是精确的 `x.y.z`。这需要在后续通过设置项进一步控制。

### 11.2 包管理器差异

不同项目可能使用 npm、pnpm、Yarn、bun。它们的升级命令不同，所以第一版最好先限制在 npm 或“以 npm 为主，其他管理器仅识别不执行”。

### 11.3 monorepo 复杂度

如果工作区中有多个 `package.json`，就会引出“当前看哪一个”和“升级命令在哪个目录执行”的问题。第一版不建议一开始就做。

### 11.4 网络与性能

当依赖数量较多时，同时请求很多最新版本会有性能与速率限制问题，需要缓存和并发控制。

## 12. 建议的 MVP 决策

为了让第一版尽快落地，建议你先采用以下决策：

- 仅支持当前工作区根目录的一个 `package.json`
- 首先处理 `dependencies` 和 `devDependencies`
- 使用 npm registry 获取最新版本
- 升级动作先支持 npm
- UI 先使用 Tree View，不急着上 Webview

## 13. 后续需要你确认但不阻塞启动的问题

- 第一版是否只支持 npm
- 是否要同时显示 `peerDependencies`
- 升级时是否保留 `^` 前缀
- 后续是否要支持 monorepo

这些问题现在不需要全部定死，但文档已经把影响点标出来了，后面做代码时就不会突然失控。
