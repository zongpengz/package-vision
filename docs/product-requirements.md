# 产品需求文档

## 1. 项目背景

前端项目经常需要检查依赖包是否过时。常见做法通常是：

- 打开终端执行 `npm outdated`
- 回到 `package.json` 判断哪些包值得升级
- 再执行安装命令完成更新

这个流程的问题是：

- 信息分散在终端和编辑器之间
- 对新手不够直观
- 查看与升级是割裂的
- 当依赖很多时，很难快速筛出“真正需要关注的那批包”

Package Vision 想解决的是：在 VS Code 里完成依赖版本巡检、筛选和局部升级。

## 2. 产品定位

Package Vision 是一个面向前端开发者的 VS Code 扩展。它提供一个位于 Activity Bar 的独立入口，在侧边栏中展示依赖的声明版本、最新版本和状态，并允许用户在编辑器里直接完成单个依赖升级。

它不是一个包管理器替代品，而是一个依赖可视化与升级入口增强工具。

## 3. 目标用户

- 维护 React、Vue、Vite、Next.js、Nuxt 等前端项目的开发者
- 想减少终端切换频率的开发者
- 希望通过真实项目学习 VS Code 扩展开发的新手

## 4. 核心问题

用户希望在不离开 VS Code 的情况下，快速知道：

- 当前工作区里有哪些依赖
- 每个依赖在 `package.json` 中声明的版本是什么
- npm registry 上的最新稳定版本是什么
- 哪些包已经过时，哪些查询失败，哪些无法比较
- 如何只升级一个依赖，而不是一次性改整个项目
- 升级后版本范围应不应该保留 `^`、`~` 或写成精确版本

## 5. 产品目标与非目标

### 5.1 当前产品目标

- 在编辑器左侧提供清晰稳定的功能入口
- 让依赖版本状态一眼可见
- 让依赖可以按状态快速筛选
- 让单个依赖升级动作可点击、可确认、可反馈
- 让升级后的版本范围策略可配置
- 让实现过程适合新手逐步学习和扩展

### 5.2 当前非目标

当前版本暂不追求以下能力：

- 自动分析 breaking changes
- 自动聚合 changelog 或 release notes
- 一次升级全部依赖
- 完整支持 `peerDependencies`、`optionalDependencies`
- 完整支持所有 workspace 协议和 monorepo 细节
- 对私有 registry、代理、认证场景做深度适配

## 6. 名词定义

- 声明版本：`package.json` 中写下的版本范围，例如 `^18.3.1`
- 最新版本：注册表中 `latest` dist-tag 指向的最新稳定版本
- 过时：最新版本不再被当前声明版本覆盖，或当前声明版本明显落后
- 升级：执行包管理器命令并更新 `package.json` 与锁文件
- 筛选：按依赖状态缩小列表范围，例如“仅显示查询失败”

说明：当前项目的核心关注点是“声明版本”和“最新版本”。如果以后要更接近 `npm outdated`，可以再增加“已安装版本”和“wanted 版本”。

## 7. 当前 MVP 功能范围

### 7.1 已实现能力

- 在 Activity Bar 中增加 `Package Vision` 入口
- 识别当前工作区中的一个或多个 `package.json`
- 解析以下依赖字段：
  - `dependencies`
  - `devDependencies`
- 在侧边栏中按 package 和依赖分组展示列表
- 每个依赖展示：
  - 包名
  - 声明版本
  - 最新版本
  - 状态标识
- 支持手动刷新版本信息
- 支持升级单个依赖到最新版本
- 支持按状态快速筛选依赖
- 支持配置升级后的版本范围写回策略
- 升级成功后自动刷新视图
- 提供输出日志和错误反馈
- 支持 `npm`、`pnpm`、`yarn`、`bun`

### 7.2 推荐但可延后

- 支持 `peerDependencies`
- 支持 `optionalDependencies`
- 支持显示 installed / wanted 版本
- 支持更细粒度的筛选与排序
- 支持 changelog 或 release notes 入口
- 支持更多设置项，例如默认展开策略

### 7.3 后续迭代

- 支持 `升级全部`
- 支持更丰富的 monorepo 展示与统计
- 支持 workspace 协议的更精细处理
- 支持私有 registry 和认证场景

## 8. 用户故事

### 8.1 查看依赖状态

作为一个前端开发者，我希望打开左侧 `Package Vision` 之后，就能看到依赖的声明版本和最新版本，这样我不用来回切终端。

### 8.2 刷新版本信息

作为一个用户，我希望点击刷新按钮后重新获取最新版本信息，这样我能确认列表不是旧数据。

### 8.3 升级单个依赖

作为一个用户，我希望对某个依赖直接点击升级，这样我能局部更新依赖，而不是一次性修改整个项目。

### 8.4 无项目时得到反馈

作为一个用户，如果当前工作区没有 `package.json`，我希望看到明确提示，而不是一个空白视图。

### 8.5 快速筛选不同状态的依赖

作为一个用户，我希望按“过时”“失败”“无法比较”“升级中”等状态快速过滤列表，这样我能更快定位需要关注的依赖。

### 8.6 控制升级后的版本写法

作为一个用户，我希望决定升级后写回 `package.json` 的版本范围是保留原风格、统一使用 `^`、`~` 还是精确版本，这样插件能符合团队规范。

## 9. 交互要求

### 9.1 入口位置

- 入口必须出现在 Activity Bar
- 视觉上与 Explorer、Search、Extensions 同级
- 点击后在主侧边栏打开扩展视图

### 9.2 依赖列表

当前实现采用 Tree View：

- 多 package 项目：
  - 第一层：package manifest
  - 第二层：依赖分组
  - 第三层：具体包项
- 单 package 项目：
  - 第一层：依赖分组
  - 第二层：具体包项

每个包项展示：

- `label`：包名
- `description`：声明版本和最新版本
- `tooltip`：package、位置、状态、升级提示
- `iconPath`：状态图标

### 9.3 升级动作

升级动作当前通过以下方式暴露：

- 包项点击
- 包项悬浮行内图标
- 包项右键菜单

升级前会弹出确认框，说明：

- 包名
- 当前声明版本
- 即将升级到的目标版本
- 可能修改 `package.json` 和锁文件

当存在更高 major 版本时，产品还需要支持：

- 默认按设置项决定是先询问、优先保守升级，还是直接升级到最新 major
- 用户通过悬浮行内图标或右键菜单显式执行“升级到最新 major”

### 9.4 快速筛选

- 视图标题栏提供筛选按钮
- 支持按状态切换：
  - 全部
  - 过时
  - 已最新
  - 查询失败
  - 无法比较
  - 升级中
- 有筛选条件时，视图应提供当前筛选提示和清除入口

## 10. 功能验收标准

### 10.1 入口与视图

- 安装扩展后，Activity Bar 中能看到扩展图标
- 点击后能打开 `Package Vision` 视图

### 10.2 数据展示

- 工作区存在一个或多个 `package.json` 时能显示依赖
- 每个依赖能看到包名、声明版本、最新版本
- 不同状态有清晰可辨识的图标与文案

### 10.3 刷新能力

- 执行刷新命令后，界面能重新获取最新版本
- 失败时有错误提示，不会导致视图失效

### 10.4 升级能力

- 对单个依赖执行升级后，命令能被正确调用
- 升级成功后，`package.json` 或锁文件发生预期变化
- 升级后的版本范围写回符合当前设置项
- 视图自动刷新为最新状态

### 10.5 筛选与配置

- 切换筛选条件后，列表只展示匹配状态的依赖
- 没有匹配结果时，显示明确空状态提示
- 修改版本范围设置后，下一次升级结果符合设置预期

### 10.6 空状态与异常

- 没有工作区时显示提示
- 没有 `package.json` 时显示提示
- 网络失败时显示错误信息
- 包管理器不可用时显示明确说明

## 11. 约束与风险

### 11.1 版本语义风险

不同包管理器默认保存策略并不完全一致。当前实现已经提供 `preserve`、`caret`、`tilde`、`exact` 四种写回策略，但对于复杂版本声明仍可能需要回退到默认策略。

### 11.2 包管理器差异

当前实现已经支持 `npm`、`pnpm`、`yarn`、`bun` 的单包升级，但它们在 monorepo、workspace、锁文件同步上的细节差异仍然较大。

### 11.3 monorepo 复杂度

当前实现已经支持多个 `package.json` 和常见 monorepo 场景，但 workspace 协议、复杂目录布局和更细粒度展示仍有继续打磨空间。

### 11.4 网络与性能

当依赖数量较多时，同时请求很多最新版本会有性能与速率限制问题，因此需要缓存和并发控制。

## 12. 当前 MVP 决策

当前仓库的 MVP 已经采用以下决策：

- 支持当前工作区中的多个 `package.json`
- 先处理 `dependencies` 和 `devDependencies`
- 使用 npm registry 获取最新版本
- 升级动作支持 `npm`、`pnpm`、`yarn`、`bun`
- UI 使用 Tree View，不急着上 Webview
- 提供快速筛选和版本范围设置项

## 13. 后续仍需确认的问题

- 是否要同时显示 `peerDependencies`
- 是否要支持 `optionalDependencies`
- 是否要支持“升级全部”
- 是否要显示 changelog / release notes

这些问题现在不阻塞开发，但继续演进时需要尽早定边界。
